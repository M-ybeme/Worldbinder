# Changelog

All notable changes to this project are documented here. Format follows [Keep a Changelog](https://keepachangelog.com/en/1.1.0/); versioning follows [Semantic Versioning](https://semver.org/) — a pre-1.0 project, so `0.MINOR.PATCH` bumps for any user-visible or structural change, `PATCH` for fixes with no scope change.

Every push to `main` should add an entry here. This is meant to be an honest record of what actually shipped, not a restatement of the roadmap's aspirations — if something was attempted and reverted, or shipped partially, say so.

## [0.11.0] - 2026-07-13

### Added

- **Milestone 10 — Maps.** Map CRUD, map image upload (reusing Milestone 9's attachment pipeline), layers, pin placement/drag/delete, location links to any entity type, visibility controls, client-side layer/type filters, and an accessible pin list — both API and web UI.
- New schema: `maps` (`image_attachment_id` a direct nullable FK to `attachments`, `on delete set null` — same shape as `campaigns.cover_attachment_id`, not a `resource_attachments` join, since a map has at most one background image), `map_layers` (`display_order`, per-layer `visibility`), and `map_pins` (`x_normalized`/`y_normalized` as `double precision`, both constrained to `[0, 1]` at the Zod boundary so pins stay positioned across responsive image sizes). **`maps` hard-deletes** (no `deleted_at` column, unlike entities/sessions/plot_threads — matches the roadmap's literal §9.12 column list) and cascades to its layers and pins via FK; deleting a layer only `set null`s its pins' `layer_id`, ungrouping rather than deleting them. `map_pins.location_entity_id` accepts **any** entity type, not just `entityType === 'location'` — the column name describes "the location on the map," not a type constraint, and ui-ux.md explicitly lists Characters/Factions/Events/Quests as valid pin subjects.
- **Four-way visibility composition prevents a specific leak**: a pin is visible to a viewer only if the map, its own layer (if any), the pin itself, _and_ its linked entity (if any) all pass that viewer's visibility check — `MapsService.isPinVisible()`, a pure exported function, mirrors Milestone 7's `SearchService.searchRelationships` "no anchor entity, so every side must be checked explicitly" reasoning. Without this, a `public` pin on a `public` map with no layer could otherwise leak a `gm_only` entity's existence and map location to a player. A pin whose linked entity has been soft-deleted is hidden from everyone, including the GM. `MapsService.getById()`/`list()` 404 (never 403) a `gm_only` map to a non-privileged viewer, per ADR-0008.
- `maps` module (`apps/api/src/maps/`): one `MapsService` owns `maps`/`mapLayers`/`mapPins` (same "domain plus its join-ish tables in one service" shape as `PlotThreadsService`), gated by a new `CampaignPolicyService.canManageMaps` (owner/gm/editor, same actor set as `canManageAttachments`/`canManagePlotThreads`). The map-image `PATCH` validates a `ready`, same-campaign, image-typed attachment via a direct `attachments` table read — no `AttachmentsModule` import, same precedent as `CampaignsService`'s cover-image field. A narrow `PATCH .../pins/:pinId/position` endpoint backs pin dragging specifically — it accepts only the two coordinate fields, so a fast-firing drag update can never clobber a concurrent label/visibility edit from the full pin-edit form. No optimistic-concurrency check on maps/layers/pins (not a listed exit criterion for this milestone, and pins are frequently-repositioned data better served by last-write-wins). Deleting a map records a `campaign_audit_events` `destructive_action` row, same precedent as attachment deletion.
- Frontend `features/maps/`: `MapListPage`/`MapFormPage` (name/description/visibility plus the same upload-then-poll-until-ready-then-PATCH image flow `CampaignSettingsPage`'s cover image already uses) and `MapDetailPage`, which composes `MapCanvas` (the map image in a container whose CSS `aspect-ratio` is set from the attachment's stored width/height, so normalized-coordinate pins stay pixel-aligned to the same map feature at any viewport width with no resize-observer logic), `MapLayerToggles` (client-side layer/unlayered filters driving both the canvas and the list from one shared state), `AccessiblePinList` (a plain keyboard-reachable button list doing the same activation as clicking a visual pin — satisfies the "keyboard users can access all pin content" exit criterion without a parallel accessibility tree over the absolutely-positioned canvas), and `MapPinForm` (inline, no modal primitive exists in this codebase yet — same precedent as `RevisionHistoryPanel`). A GM/editor-only "manage mode" toggle governs the interaction model: view mode clicking a linked pin navigates to its entity page (ui-ux.md: "not a giant popup"; a freestanding label-only pin is inert), manage mode clicking opens the inline edit form and dragging (pointer events + `setPointerCapture`, with a small movement threshold to distinguish a click from a drag) repositions. "Maps" added to `CampaignLayout`'s nav between Threads and Search, matching the roadmap's literal §9 nav ordering.
- 6 new unit tests (`isPinVisible`'s truth table, including the public-pin/gm_only-entity leak case) and 1 new policy spec case (`canManageMaps`). 16 new integration tests (`maps.e2e-spec.ts`) covering map/layer/pin CRUD, cross-campaign rejection on all three, delete-cascade behavior (map delete removes layers/pins and audits; layer delete ungroups rather than deletes pins), the reposition endpoint's narrow field scope and range rejection, the full visibility-leak matrix (`gm_only` map, `gm_only` layer under a public map, a public pin linked to a `gm_only` entity, and a soft-deleted linked entity), and the map-image attachment gate's not-ready/cross-campaign/non-image rejections.
- Manually verified end to end against the running dev stack (Playwright-driven): created a map, uploaded an image and watched it move through pending → processing → ready via polling with correct aspect-ratio rendering; placed both an entity-linked pin and a freestanding labeled pin by clicking the canvas; dragged a pin to a new position; confirmed the accessible pin list mirrors the canvas exactly, including after toggling the "Cities" layer off (both surfaces hid the linked pin simultaneously); confirmed clicking a linked pin in view mode navigated to its entity page; set the map to `gm_only`, invited and accepted a real player-role member through the full registration/verification/invitation flow, and confirmed the map is absent from that player's map list and returns "This map could not be loaded" (a 404, not a 403) on direct navigation to its URL.
- **Scope notes**: no optimistic concurrency on maps/layers/pins (see above). Filters are client-side only — no new server query parameters, same reasoning as `PlotThreadPicker`. ui-ux.md's "pins can link to plot threads/session references/attachments" is narrower in this milestone than the aspirational UX sketch — the roadmap's literal §9.12 schema gives `map_pins` a single `location_entity_id` FK, so v1 pins link to at most one entity; no separate pin-to-plot-thread or pin-to-attachment link exists yet.

## [0.10.0] - 2026-07-12

### Added

- **Milestone 9 — Attachments.** Presigned uploads to S3-compatible storage (MinIO locally), a background worker that authoritatively verifies file type via hand-rolled magic-byte detection, computes a sha256 hash and image dimensions, a resource-attachment picker on entity/session/plot-thread detail pages, campaign cover images, short-lived signed downloads, and an hourly cleanup sweep for abandoned uploads — both API, worker, and web UI. This is the first milestone to give `apps/worker` real job-processing work (it was an idle scaffold since Milestone 0) and the first to add BullMQ or an S3 client anywhere in the monorepo.
- New schema: `attachments` (`storageKey` fully opaque — `attachments/{campaignId}/{attachmentId}`, no user-controlled filename/extension in the key at all; `uploadedByUserId` nullable with `onDelete: 'set null'`, deliberately not copying `entities.createdByUserId`'s cascade pattern — a cascaded user deletion would delete the row at the FK level with no application code running, so the cleanup job would never fire and the underlying storage object would become permanently unreachable garbage) and `resourceAttachments` (a real many-to-many join — one attachment can legitimately be linked to more than one resource — with `resourceType` as plain `text`, not an enum: `entityWikiLinks.sourceResourceType` already established this precedent for polymorphic discriminators expected to grow, and reusing Milestone 8's `resourceRevisionTypeEnum` here would have created a real DB-level coupling between two unrelated features). Same migration adds the FK `campaigns.coverAttachmentId → attachments.id` that Milestone 2 stubbed as an unconstrained column.
- New `apps/api/src/storage/` (`StorageService`: presigned PUT/GET via `@aws-sdk/client-s3` + `@aws-sdk/s3-request-presigner`, `HeadObjectCommand`-verified completion, best-effort `DeleteObjectCommand`, `onModuleInit` bucket auto-creation outside production) and `apps/api/src/jobs/` (`AttachmentQueueService`, a BullMQ producer with its own Redis connection — `maxRetriesPerRequest: null`, distinct from the `@Global()` rate-limiting `REDIS` token — placed as a sibling of `attachments/`, not inside it, per the roadmap's own directory sketch, anticipating Milestone 12's export/import queues will reuse it). `apps/worker` gained matching `storage/`/`jobs/` modules plus `attachments/process-attachment.ts` and `attachments/cleanup-sweep.ts` — both plain exported async functions (dependency-injected `pg.Pool` + `S3Client`, not framework DI), callable both by the real BullMQ `Worker` and directly by integration tests, so `pnpm test:integration` never needs a live queue consumer running.
- **Hand-rolled magic-byte sniffing instead of the `file-type` npm package**: `apps/api` and `apps/worker` both compile to CommonJS (no `"type": "module"`), and `file-type` has been ESM-only since v17. PNG/JPEG/GIF/WebP/PDF each have trivial fixed signatures, so `packages/validation/src/attachment-detection.ts` implements `sniffMimeType()` directly — sidesteps the dependency entirely and is trivially unit-testable with fixture buffers. Plain text and Markdown have no magic number by design, in real tension with "do not trust extension" (roadmap §16.2) — mitigated by a `looksLikeText()` heuristic (null-byte and control-byte-ratio checks on the first 8KB); both collapse to a single detected `text/plain` value server-side, since Markdown-vs-plain-text cannot be verified any other way. `image-size` (a real dependency, not hand-rolled — per-format dimension parsing is intricate enough to justify one) extracts width/height for image types.
- **The worker's processing is asynchronous, and the API surface is built around that, not around pretending it isn't**: `complete()` verifies the object actually landed in storage (`HeadObjectCommand`) and enqueues processing, but does not wait for it — `link()` and the cover-image `PATCH` both require `status: 'ready'` and reject with `409`/`403` otherwise. A new `GET /campaigns/:campaignId/attachments/:attachmentId` status-check endpoint backs the frontend's poll-until-ready step between upload and link — this exact race (the frontend originally called `link()` immediately after `complete()`, before the worker had finished) was caught via live browser testing, not by any automated test, and fixed by adding the poll rather than by weakening `link()`'s `ready`-only guard.
- **Download authorization is resource-scoped by construction, not by a fallback rule.** `GET /campaigns/:campaignId/attachments/:resourceType/:resourceId` lists attachments linked to _that_ resource, gated purely by that resource's current live visibility — no `@RequireCampaignRole` guard, the same shape as Milestone 8's `RevisionsController.list()`. A viewer who can see resource A sees A's attachment list regardless of whether the same attachment is also linked to a `gm_only` resource B, because they never issue a request in B's context — the "does visibility need to check any-vs-all linked resources" question this design could have run into never has to be answered in code. Each entry carries a freshly generated, short-lived (~15min) signed download URL embedded directly in the response.
- Campaign cover images extend the _existing_ `PATCH /campaigns/:campaignId` (`coverAttachmentId`, validated via a direct `attachments` table read inside `CampaignsService` — matching its dashboard method's existing "read a sibling table directly" precedent — rather than importing `AttachmentsService`) instead of a new dedicated endpoint. Exposed to every active member regardless of role, the same way a campaign's name/description already has no visibility gate.
- `CampaignPolicyService.canManageAttachments` (owner/gm/editor, same actor set as `canEditEntities`/`canManagePlotThreads`, kept as its own method per the existing "distinct capability that could diverge later" convention).
- New `packages/ui/src/FileDropzone.tsx` (click-to-browse + drag-and-drop; no file-input primitive existed in this package before). New `apps/web/src/features/attachments/`: an `AttachmentsPanel` mirroring `RevisionHistoryPanel`'s self-contained shape exactly, wired into the entity/session/plot-thread detail pages between "Plot Threads" and "Backlinks"/"Revision History" per `docs/planning/ui-ux.md`'s layout sketch; a dedicated `uploadToStorage()` helper that deliberately bypasses `apiClient.ts` (no `Authorization`/`credentials` sent to the storage origin); the first `refetchInterval` polling precedent in this codebase, keeping the panel live while the worker processes an upload. A small cover-image control on `CampaignSettingsPage`.
- 21 new worker unit tests (`sniffMimeType`/`looksLikeText` fixture buffers covering every allowed type plus rejection cases; `processAttachment()` and `cleanupAbandonedAttachments()` with injected fake `pg.Pool`/`S3Client`, no real infrastructure needed) and 1 new policy spec case (`canManageAttachments`). 18 new integration tests (`attachments.e2e-spec.ts`) running the real presign → PUT-to-MinIO → complete pipeline end to end (not mocked), plus permission gates, the not-ready/cross-campaign link rejections, resource-scoped visibility (a `gm_only`-linked attachment invisible to a player), the unlinked-attachments management list, delete + its `destructive_action` audit event, and all three cover-image validation paths. The worker's own file-processing correctness (magic bytes, hashing, dimensions) is covered by its unit tests with fakes rather than duplicated in the API's integration suite, which instead verifies the API surface built on top of the resulting `ready` state.
- Manually verified in a real browser (Playwright-driven): uploaded an image to an entity's Attachments panel and watched it move through the pending → processing → ready states via polling, confirmed the thumbnail and signed download URL render correctly; uploaded a campaign cover image through the same upload → poll → PATCH flow and confirmed it's visible to a player-role member. Also manually verified end to end via curl against a live worker process: the full pipeline (including a genuinely corrupt "PNG" that passes the magic-byte check but fails `image-size` parsing, correctly rejected) and the cleanup sweep (a backdated `pending` row correctly rejected and its storage object removed).
- **Scope notes**: `attachments.visibility` is schema-complete (per the roadmap's literal column list) but not enforced by any v1 read path — no UI surface reaches an attachment with zero `resource_attachments` links other than the cover-image flow, which deliberately doesn't gate on it either (campaigns have no visibility concept). 20MB size ceiling and the 24h-abandonment/hourly-cadence cleanup sweep are documented judgment calls, same spirit as Milestone 8's `REVISION_WINDOW_MINUTES`. Animated-GIF detection is out of scope (roadmap says "optionally static-only"). No automated integration test exercises the cleanup sweep against real Postgres (it's covered by unit tests with fakes plus a manual live run) — adding a live-worker-process integration test was judged not worth the added test-environment complexity for this milestone.

## [0.9.0] - 2026-07-12

### Added

- **Milestone 8 — Revisions and Audit.** Immutable revision snapshots for entities, sessions, and plot threads with autosave-window grouping, a revision-history panel with field-level diff and one-click restore, and a new campaign-scoped audit/activity log covering role changes, member removal, content reveal, revision restore, and destructive actions — both API and web UI.
- New schema: `resource_revisions` (`resourceType` enum of `entity`/`session`/`plot_thread`, full GM-inclusive `snapshot_json`, `revision_number`, nullable `change_summary`, `created_by_user_id`) and `campaign_audit_events` (`type` enum, nullable `actor_user_id`/`target_resource_type`/`target_resource_id`, structured `metadata_json` only — never content bodies, per roadmap §11.14). No update/delete method exists anywhere on `CampaignAuditService`, and no controller mutation route was ever added — that omission is the entire mechanism behind the "audit records cannot be edited through normal APIs" exit criterion, not a permission check that could be bypassed.
- **Two dependency-free leaf modules avoid a circular-dependency dead end.** Recording a revision must happen _inside_ `EntitiesService`/`SessionsService`/`PlotThreadsService`'s `create()`/`update()`; restoring one must replay the _full_ owning-service write path (not a raw `UPDATE` that would skip tag sync, wiki-link refresh, join-table resync, or tsvector rebuild) — the opposite dependency direction. `RevisionRecorderModule`/`RevisionRecorderService` (only `recordRevision()`, only `@Inject(DRIZZLE)`) and `CampaignAuditModule`/`CampaignAuditService` (`record()`/`list()` only) are the leaves that `EntitiesModule`/`SessionsModule`/`PlotThreadsModule`/`MembershipModule`/`CampaignsModule` import one-directionally to write; `RevisionsModule` and `CampaignAuditViewModule` are separate orchestrator modules that import the owning modules the other way, to list/restore. No `forwardRef()` anywhere (still unused in this codebase). Restore-mapping logic (`entitySnapshotToUpdateInput`/`sessionSnapshotToUpdateInput`/`plotThreadSnapshotToUpdateInput`) lives as pure functions exported next to each owning service rather than new DI surface.
- **Windowed merge grouping**: same actor + same resource + within a 30-minute window (`REVISION_WINDOW_MINUTES`, a documented judgment call in the spirit of `plot-threads.service.ts`'s `NEGLECT_THRESHOLD_SESSIONS`) updates the latest revision row's snapshot in place rather than inserting a new one — autosave-driven edits during one sitting collapse to a single revision instead of one per keystroke-triggered save. A `recordRevision()` bug where a merged row's `changeSummary` stayed stuck on a stale value (e.g. "Restored from revision #1" surviving into a later plain edit) was caught via live manual testing, not the automated suite, and fixed by making the field unconditionally overwritten on every merge.
- **Restore always creates a new, distinct revision — never merges into the restoring actor's own recent edit window.** `recordRevision()` takes an `allowMerge` flag: `true` for normal `create()`/`update()` calls, `false` when the write originates from `RevisionsService.restore()`. Restore replays the snapshot through the real owning-service `update()` (full side effects, not a raw write), reuses the live resource's current `updatedAt` so the optimistic-concurrency check is a no-op in the normal case while still catching a genuine race, and rejects with `409` if the live resource has since been soft-deleted rather than producing a confusing generic `404`.
- **Field-level diff without a diffing library** (none — `diff`/`jsdiff`/`fast-diff` — is installed in this repo, and adding one for a single read endpoint wasn't judged worth it). `RevisionsService.list()` computes plain-text extracts per content field server-side (reusing Milestone 7's `extractPlainText`) and ships them alongside the raw snapshot, so the frontend does a trivial string comparison instead of walking TipTap JSON. Scoped to a changed-field list with before/after plain text, not a word-level diff — noted as a follow-up rather than a dependency added for v1.
- **GM-content omission is enforced at read time, not write time.** Snapshots always store the full GM-inclusive shape; `RevisionsService.list()`/`restore()` redact or omit GM-only fields based on the _viewing/restoring_ actor's _current_ permission, not the snapshot's original author's permission — matching roadmap §13.1's "revision history must follow the current resource permission policy" without needing two stored variants per snapshot.
- `GET /campaigns/:campaignId/revisions/:resourceType/:resourceId` and `POST /campaigns/:campaignId/revisions/:revisionId/restore` (both in the roadmap's own route table); `GET /campaigns/:campaignId/audit` (owner/gm-gated, the first _read_ route in this codebase to use `@RequireCampaignRole` — not in the roadmap's literal route table, which is representative rather than exhaustive, but structurally necessary to back the required audit activity view).
- Frontend: one shared `RevisionHistoryPanel` (`features/revisions/`) reused identically on the entity/session/plot-thread detail pages — same self-contained, own-data-fetching shape as Milestone 4's `RelatedContentPanel`. Restore uses `window.confirm(...)`, matching every other confirmation in this codebase (no modal component exists in `packages/ui` yet, and this wasn't the milestone to add the first one). A new `AuditPage` (`features/audit/`) at `/app/campaign/:id/audit`, linked from a new "Activity log" section on the owner/gm-gated Settings page rather than the roadmap's fixed primary nav (which has no slot for it).
- 7 new unit tests (`shouldMergeRevision`'s merge-vs-insert boundary, factored out of `RevisionRecorderService` the same way `isNeglected`/`checkRelationshipTypeCompatibility` were) and 15 new integration tests (`revisions.e2e-spec.ts`, `audit.e2e-spec.ts`) covering merge-window grouping, restore always producing a distinct revision even immediately after the restoring actor's own edit, cross-campaign spoofing rejection on both list and restore, soft-deleted-resource restore rejection while history stays listable, GM-content omission on both revision snapshots and hidden-resource history, and audit events recorded for role change, member removal, content reveal, revision restore, campaign archive, and destructive deletion.
- Manually verified in a real browser (Playwright-driven, per this repo's UI-verification convention): a GM and a second "editor" account produced two distinct, differently-authored revisions on an entity; the revision-history panel listed both with correct actor/timestamp, "View changes" expanded a field-level diff (summary and tags), and restoring an older revision reverted the live content and appended a new, unmerged revision rather than silently folding into anything. The Settings page's new "Activity log" link correctly reached `/audit`, which rendered the recorded `revision_restored` event.
- **Scope notes**: relationships are excluded from revisions (a small structured link record, not long-form content — matches how relationship edits are already treated as single-field changes elsewhere); no revision snapshot is taken on delete (soft-deleting a resource doesn't itself create a final revision); login/password/session security events stay on the existing auth-only `security_events`/`AuditService` and aren't duplicated here; campaign export/import audit events are deferred (Milestones 12/13 don't exist yet).

## [0.8.0] - 2026-07-12

### Added

- **Milestone 7 — Search.** Full-text search across entities, sessions, plot threads, and relationship descriptions, with weighted ranking, `pg_trgm` fuzzy/prefix name matching, a global `Ctrl`/`Cmd+K` search overlay, a dedicated results page with type filters and pagination, keyboard navigation, and highlighted snippets — both API and web UI.
- New schema: `search_vector_public`/`search_vector_gm` `tsvector` columns on `entities`, `sessions`, and `plot_threads` (via a shared `customType` helper — no native tsvector column type in drizzle-orm), plus a single `search_vector` column on `entity_relationships` (weight D, description only — no public/GM split exists on that table). **Two vector columns per table, not one gated at query time**: `searchVectorGm` additionally folds in secret content (`gmContentJson`/`plannedContentJson`), so a non-GM viewer's query can never match against hidden text — the same field-omission stance this codebase already takes on `gmContentJson` itself, applied one layer earlier so it can't leak through result _presence_, not just result _content_. Row-level `visibility` stays a separate, still-mandatory filter on top of this — the vector split only gates content within an otherwise-visible row. New GIN indexes on every tsvector column, `gin_trgm_ops` trigram indexes on each table's name/title column, and a leading `campaign_id` index on all four tables (none existed before — Postgres doesn't auto-index FK columns, and none of the four search-touched tables had ever needed one until now).
- **Documented migration deviation**: enabling `pg_trgm` requires one `CREATE EXTENSION IF NOT EXISTS pg_trgm;` statement that `drizzle-kit generate` cannot produce from `schema.ts`, and `infrastructure/migrations/` is hook-blocked from direct edits by design. Per the `db-migration` skill's own guidance for cases drizzle can't express, this one line was added by hand to `infrastructure/migrations/0007_flawless_impossible_man.sql` outside the normal generate-only flow — everything else in that migration is machine-generated.
- `search` module (`apps/api/src/search/`): `search-vector.util.ts` (`extractPlainText` — a new TipTap-JSON-to-plain-text walker, modeled on `WikiLinksService.extractMentions`'s recursion shape; `buildWeightedTsvector` — builds the `setweight(to_tsvector(...), 'A'..'D') || ...` SQL expression used at every write site). `SearchService` follows the `campaigns.service.ts` dashboard's precedent exactly: injects only `DRIZZLE` + `CampaignPolicyService` (no sibling services), runs four independent per-table queries rather than a UNION (nothing in this codebase uses UNION), and merges/re-ranks the results in JS. Each query computes a ranking tier (roadmap §14.3: exact name → exact alias → name prefix → fuzzy name → tag/summary → content → relationship description) and a score via `ts_rank_cd`/`similarity()`/`ilike`, filtered by the same conditional-visibility-array pattern every other list query already uses. The relationship query joins `entities` aliased twice (source/target) and requires **all three** of the relationship's own visibility and both endpoints' visibility to pass — stricter than `RelationshipsService.canSeeRow`'s one-sided check, which is safe there only because it's always called from an already-visible anchor entity; global search has no anchor, so both sides must be checked explicitly in SQL (roadmap §14.5: filter inside the query, never rank-then-drop).
- Snippets are built from plain-text substring matching with recorded highlight offsets (`{ text, highlights: [number, number][] }`), not `ts_headline()` — this codebase doesn't persist raw plain-text bodies anywhere (only the derived tsvector), and running a JS substring scan over the handful of already-fetched, already-`LIMIT`-ed result rows is cheaper than adding storage just to feed `ts_headline`. Offsets only, never embedded markup, so the frontend renders via plain text nodes wrapped in `<mark>` — no `dangerouslySetInnerHTML` anywhere in this feature, consistent with the rest of the app.
- `GET /campaigns/:campaignId/search` (new `SearchController`, guarded the same as every other campaign-scoped route) serves both the overlay (small `limit`) and the results page (paginated, `types`-filtered) from one endpoint, matching the roadmap's single documented route. Plain `limit`/`offset`, not `packages/validation`'s existing `cursorPaginationQuerySchema` — that schema has zero call sites anywhere in the repo and doesn't fit a JS-merged, re-ranked result set from four independent queries.
- Frontend `features/search/`: `SearchOverlay` (portal-rendered modal, 200ms-debounced query matching `EntityPicker`'s existing debounce pattern, arrow-key nav modeled on `packages/ui/src/Combobox.tsx`'s modulo-wrap handling — referenced as a pattern, not reused, since `Combobox` is a single-input-bound field, not a multi-type modal), `SearchResultsPage` (URL-driven `q`/`types`/`page`, matching the roadmap's "search terms live in the URL" state-ownership rule), and a new `useSearchOverlayStore` Zustand store holding overlay open/query state only. `CampaignLayout` gained a "Search" nav link and the `Ctrl`/`Cmd+K` keydown listener. Relationship results have no standalone page to link to — they navigate to the source entity's detail page instead (same scope shape as Milestone 4's "relationships surface as panels, not their own pages" decision).
- **Scope note**: Timeline events (roadmap §14.1) are not part of this milestone's search coverage — no timeline/calendar module exists yet (Milestone 11), so there's nothing to index. Same honest-scope-note spirit as Milestone 4's "no `/world/relationships` graph view this milestone."
- 9 new unit tests (`extractPlainText`, `buildWeightedTsvector`, and `searchQuerySchema` — the explicit §20.1 "Search query parsing" unit-test target) and 9 new integration tests (`search.e2e-spec.ts`) covering ranking-tier ordering, cross-campaign isolation, hidden-entity exclusion, hidden-relationship exclusion via the two-sided endpoint check, GM-only content never matching a player's query even when the parent entity is public, and pagination/type-filtering.
- New `apps/web/e2e/search.spec.ts`: a GM finds both a public and a GM-only entity (and a phrase buried in GM-only content) via the overlay and the results page; an invited player finds the public entity by name but never the GM-only entity or the secret phrase, through either surface — the roadmap §20.4 step 14 ("Search visible content") E2E target.
- **Performance validation**: new `apps/api/src/database/seed-perf.ts` (`pnpm db:seed:perf`) batch-inserts a 10k-entity/50k-relationship/200-session/2k-plot-thread campaign directly via Drizzle — bypassing the service layer for speed, but building tsvectors through the same `search-vector.util.ts` functions the real services use, so the fixture matches production writes rather than a special-cased backfill. New `apps/api/src/search/search-benchmark.ts` (`pnpm search:benchmark`) runs representative queries (exact/prefix/fuzzy name, content word, relationship-description word, multi-word phrase) against the seeded campaign and prints p50/p95/p99 — a plain script, not a Jest test, since a hard latency assertion against a shared local Postgres would be flaky in CI. Measured **p95 = 337ms** against the roadmap §14.4 budget of 500ms — exit criterion met. This is a scoped-down harness for this milestone's own criterion only; the full §20.6 cross-feature profiling suite (dashboard, entity details, relationship queries, exports, imports) is deferred to Milestone 14.

## [0.7.0] - 2026-07-12
